# 个性化旅游系统 - 需求与系统设计文档

> 根据《数据结构课程设计》讲义整理

## 1. 问题描述
当前大家经常利用假期去各地旅游。个性化旅游系统可以帮助大家管理自己的旅游活动，具备**旅游地点推荐、旅游路线规划、旅游场所查询、旅游日记交流**等功能。
- **旅游前**：需先按照旅游目的地的热度、评价和个人兴趣**选择旅游目的地**；
- **旅游中**：在景点（包括学校）内部也需要根据游览的目标**规划最优的参观线路**，在游览过程中还可以给出相应的景点介绍和**场所查询**；
- **旅游后**：可以根据所拍照片和游览经历生成**旅游日记**。

---

## 2. 数据要求 (硬性指标)
- **景区和校园数量**：至少200个（景区和校园内部可以一致）。
- **节点数量**：
  - 建筑物（景点、教学楼、办公楼、宿舍楼）数不少于 **20** 个。
  - 其它服务设施不少于 **10** 种（商店、饭店、洗手间、图书馆、食堂、超市、咖啡馆等），数量不少于 **50** 个。
- **图结构要求（道路图）**：包括各建筑物和服务设施，边数不能少于 **200** 条（建议爬取真实地图数据）。
- **用户数**：系统用户数不少于 **10** 人。

---

## 3. 基本功能需求与核心算法

### 3.0 系统与架构通用要求
- **创新需求**：系统需解决当前旅游系统提供不了的实际需求。
- **数据结构要求**：每个功能后的核心算法**必须基于自己设计的数据结构**，自己编程实现。
- **验证与体验**：可使用智能体辅助开发及比较算法；最终提供**真实可用**的系统且带**图形界面/导航地图**展示，开发语言/架构不限。

### 3.1 旅游推荐 (对应 成员B)
- **基本推荐**：按旅游热度、评价和个人兴趣推荐旅游景点和学校。
- **★ 核心算法（排序）**：设计**排序算法**，考虑到用户通常只看**前10个**，要求**不经过完全排序可以排好前10 (Top-K问题，建议基于最小堆 Priority Queue)**。考虑数据动态变化。
- **查询过滤**：输入名称、类别、关键字查询，并对多项结果按热度/评价排序。(**核心算法：查找算法 + 排序算法**)

### 3.2 旅游路线规划 (对应 成员A)
- **单目标路线**：输入目标点，规划从当前出发的最优路线。(**★ 核心算法：最短路径算法，如 Dijkstra 或 A***)
- **多目标路线 (TSP问题)**：输入多个目标，从当前出发，参观完**返回当前位置**。(**★ 核心算法：途经多点最短路径算法**)
- **多种路线规划策略**：
  1. **最短距离策略**：路程最短。
  2. **最短时间策略**：引入**拥挤度**(<=1的正数)，`真实速度 = 拥挤度 * 理想速度`。
  3. **交通工具策略**：校区(步行/自行车)，景区(步行/电瓶车)。电瓶车路线固定且上车即走。求**不同拥挤度下的最短时间线路（含混合交通）**。
  4. **室内导航策略**：模拟教学楼/博物馆内部（大门到电梯 -> 楼层间电梯 -> 楼梯内到房间）。

### 3.3 场所查询 (对应 成员A)
- **附近查询**：选定某点找一定范围内的超市、卫生间等。(**★ 核心算法：路径排序，注意：不能根据直线距离！必须根据道路图的实际通行距离**)
- **关键字查找**：通过类别/名称找附近的服务设施并排序。(**核心算法：查找 + 排序**)

### 3.4 旅游日记管理与交流 (对应 成员C)
- **日记管理**：
  - 用图文/视频记录，支持用户评分。浏览量即热度。
  - 日记推荐排序。(**核心算法：排序算法**)
- **日记搜索与交流**：
  - **按目的地查询交集**：(**核心算法：查找算法和排序算法**)
  - **名称精确查询**：考虑数据量大和变化快，需高效。(建议：**哈希表 Hash 或 平衡二叉树 AVL/红黑树**)
  - **★ 全文检索**：按日记内容搜索。(**核心算法为文本搜索，如 KMP / 字典树 Trie / 倒排索引**)
  - **★ 数据压缩**：对旅游日记进行压缩存储。(**核心算法：无损压缩，如 Huffman 编码**)
  - **AIGC创新**：调用大模型API，根据照片、文字描述等生成动画。

### 3.5 美食推荐 (对应 成员B)
- **精准推荐**：按热门、评价和距离排序，并按**菜系**过滤。(**核心算法为排序，要求同上：不完全排序排好前10，Top-K**)
- **模糊查找**：输入名称、菜系、饭店进行基于内容的模糊查找及综合排序。(**核心算法：模糊查找算法 + 排序算法**)
